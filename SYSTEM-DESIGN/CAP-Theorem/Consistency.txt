=> Types of Consistency (Very Important)
    Consistency defines how up-to-date and ordered data appears across distributed systems. Strong consistency guarantees latest data at the cost of 
    availability, while eventual consistency maximizes availability by allowing temporary inconsistencies. Between these extremes lie practical models like 
    causal, session, and monotonic consistency, which balance correctness and performance based on application needs.

    1ï¸âƒ£ Strong Consistency (Linearizability)
    After a successful write, all subsequent reads anywhere in the system return the latest value. The system behaves as if there is one single copy of data 
    and one global timeline.
        Guarantee:
        -> Reads always return latest write
        -> Global order of operations
        How achieved:
        -> Single leader
        -> Synchronous replication
        -> Majority quorum
        Cost:
        -> High latency
        -> Lower availability
        Used in:
        -> Banking
        -> Inventory
        -> Payments
    2ï¸âƒ£ Eventual Consistency
    If no new updates are made, all replicas will eventually converge. Eventual consistency prioritizes availability and scalability, allowing temporary 
    inconsistencies that resolve over time.
        Guarantee:
        -> If no new writes, all replicas converge eventually
        Reality:
        -> Temporary stale reads allowed
        Benefits:
        -> High availability
        -> Low latency
        -> Partition tolerant
        Used in:
        -> Social feeds
        -> Likes
        -> Analytics
    3ï¸âƒ£ Causal Consistency
    Causally related operations are seen in the same order, but unrelated operations may be seen in different orders.
    Guarantee:
    -> Related operations maintain order
    -> Unrelated operations may be seen out of order
